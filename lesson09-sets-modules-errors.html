Part 1 — Sets

What is a Set? 🎱

🛍️

Real life: A bag of marbles

Imagine a bag where every marble colour is unique — you'd remove duplicates. Order doesn't matter; you just care which colours are in the bag.

R
B
G
Y
P

No duplicates. No "first" marble.

🐍

In Python

A set is created with curly braces {}. It stores items with NO duplicates and NO fixed order.

# Creating a set colors = {"red", "blue", "green"} # Check membership — super fast! print("red" in colors) # True print("pink" in colors) # False # Add a new item colors.add("yellow") print(colors)
⚡

Three key properties

1. No duplicates

Adding "red" twice keeps only one "red"

2. No fixed order

Can't do colors[0] — sets have no index

3. Fast membership test

"red" in colors is extremely fast

Sets — No Duplicates

Sets always keep items unique ✨

What happens when you add duplicates to a set:

nums = {3, 1, 4, 1, 5, 9, 2, 6, 5} print(nums) # {1, 2, 3, 4, 5, 6, 9} ← duplicates gone! tags = {"python", "python", "coding"} print(tags) # {"python", "coding"} ← only one "python" # A common trick: remove duplicates from a list votes = ["Alice","Bob","Alice","Carol","Bob"] unique_voters = set(votes) print(unique_voters) # {"Alice","Bob","Carol"}

💡 Pro Trick: Wrap a list with set() to instantly remove all duplicates!

Sets — Methods

Working with sets in Python 🛠️

# Create a set s = {"cat", "dog", "bird"} # Add an item s.add("fish") print(s) # Remove — raises error if not found s.remove("bird") # Discard — safe! no error if missing s.discard("elephant") # no crash # Check membership print("cat" in s) # True print(len(s)) # number of items # Loop through a set for animal in s: print(animal)

Operations with symbols

a = {1,2,3} b = {2,3,4} a | b # Union: {1,2,3,4} a & b # Intersect: {2,3} a - b # Diff A-B: {1} b - a # Diff B-A: {4}

⚠️ Important rules

  • Sets use {} but an empty set must be set() not {} — that creates an empty dict!
  • Items inside a set must be immutable — strings, numbers, tuples ✅ — but NOT lists ❌
  • You can't access set items by index: s[0] will crash

Part 3 — Error Handling

When programs go wrong 💥

Errors (called exceptions) crash your program. Python tells you exactly what went wrong — learn to read the message!

SyntaxError
Code was written incorrectly — Python can't even read it
if x = 5: ← should be ==
TypeError
Wrong type for an operation
"hello" + 5 ← can't add string + int
ValueError
Right type, but the value makes no sense
int("hello") ← "hello" isn't a number
NameError
Variable doesn't exist yet
print(score) ← score was never defined
ZeroDivisionError
Dividing by zero (mathematically impossible!)
10 / 0 ← math error
IndexError
Trying to access an index that doesn't exist
my_list[99] ← only 3 items in list

🎯 The goal: Instead of letting these crash your program, we can catch them and handle them gracefully using try and except!

Error Handling — Why It Matters

Why error handling? And why does it make code cleaner? 🛡️

🌍 Real programs always face surprises

⌨️A user types "abc" where a number was expected
📂A file your program needs has been deleted or moved
📡The internet connection drops mid-download
🧮A sensor sends corrupted data instead of a clean number

✅ Three reasons to always handle errors

🛡️

Robustness — your program keeps running even when something unexpected happens

📖

Readability — code clearly separates what should happen from what to do if it goes wrong

🔧

Maintainability — easy to add new error cases later without rewriting everything

📖 How error handling improves readability

😣 Without try/except — every failure needs its own if:

def divide(a, b): if not isinstance(a, (int, float)): print("a must be a number") elif not isinstance(b, (int, float)): print("b must be a number") elif b == 0: print("Can't divide by zero") else: print(a / b) # actual goal buried at the end!

😌 With try/except — intent is front and centre:

def divide(a, b): try: print(a / b) # ← what we WANT — seen first! except ZeroDivisionError: print("Can't divide by zero") except TypeError: print("Numbers only, please")

💡 The big idea: try shows the happy path — what the code is supposed to do — right at the top. Each except names exactly what could go wrong and handles it in one clear place. A reader understands the intent immediately, without hunting through a wall of ifs.

Error Handling — Basics

try and except — the safety net 🥅

😱 Without try/except — CRASH!

age = int("hello") # 💥 CRASH print("You are", age) # Program stops with: ValueError

😌 With try/except — HANDLED!

try: age = int("hello") print("You are", age) except ValueError: print("That's not a number!") # Prints: "That's not a number!" ✅

How it works — step by step

1

try: Python tries to run the code inside. If it works, great! The except block is skipped entirely.

2

except: If an error occurs in try, Python jumps here instead of crashing. You decide what to do with the problem.

3

Program continues! After the except block, the rest of your program runs normally. No more crash!

🔑 Key rule: Always name the specific error type in except (like ValueError) so you catch only what you expect. Using bare except: without a type catches everything — which can hide real bugs!

Error Handling — Advanced

Multiple except, else & finally 🔧

# Multiple except blocks def safe_divide(a, b): try: result = a / b except ZeroDivisionError: print("Can't divide by zero!") except TypeError: print("Both inputs must be numbers") else: # Runs ONLY if NO error occurred print("Answer:", result) finally: # ALWAYS runs — error or not print("Done calculating.") safe_divide(10, 2) # Answer: 5.0 # Done calculating. safe_divide(10, 0) # Can't divide by zero! # Done calculating.

Multiple except

Stack as many except blocks as you need — one for each error type. Python checks them top to bottom and runs only the FIRST matching one.

else — the "it worked!" block

The else block runs ONLY if the try block completed without any errors. Use it for code that should only run when everything is fine.

finally — the "always" block

The finally block always runs, no matter what — whether there was an error or not. Perfect for clean-up code like closing files.
