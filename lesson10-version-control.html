Part 1 — The Big Idea

What is version control? 🕹️

🎮

Like saving your game

Every time you hit a checkpoint, you can always reload an earlier save if something goes wrong. Version control gives your code those same save points.

📝

Like Google Docs history

Ever clicked "See version history" in a doc? Version control is that same idea — but built specifically for code, with superpowers added on.

🧪

The real definition

A system that records every change made to a project over time — so you can look back, compare versions, and restore any past version whenever you need to.

🐙 Git is the most popular version control tool in the world. It was actually created in 2005 by Linus Torvalds — the same person who created the Linux operating system!

Part 1 — Why It Matters

Why every software team relies on it 🎯

🤝

Team collaboration

🧪

Safe experimenting

🐞

Faster bug-fixing

Compare versions to see exactly what broke something

⏪

Real time travel

🌍

Works anywhere

🏆

An industry standard

Every professional developer is expected to know this

🎯 Bottom line: version control turns "I hope this doesn't break anything" into "if it breaks, I'll just go back."

Part 1 — Real-World Examples

You use version control's results every day 🌍

🐧

Linux

Built by thousands of volunteers worldwide, coordinated almost entirely through Git — the same tool created by Linux's founder, Linus Torvalds.

🐍

Python itself

Python's own source code lives in a Git repository on GitHub, where anyone can see every single change ever made to the language.

🏢

Every major tech company

Google, Microsoft, Amazon, Netflix — all of them use version control internally. It's considered as fundamental as having files and folders.

🎮

Video game studios

Studios track code AND art assets — characters, levels, sounds — so hundreds of developers don't overwrite each other's work.

🚀 Fun fact: NASA publishes some of its real software publicly on GitHub for anyone in the world to study, use, and even contribute to.

Part 2 — A Common Mix-Up

Git vs. GitHub — they're not the same thing! 🤔

Git 🐙GitHub 🌐
What is it? A program installed on your computer A website that hosts your repositories online
Works offline? ✅ Yes — fully ❌ No — it's a cloud service
What it does Tracks history, creates commits and branches Stores your repos online, lets you collaborate and review
Created 2005, by Linus Torvalds 2008 (now owned by Microsoft)
Alternatives? None really — it's the standard GitLab, Bitbucket also exist

💡 Analogy: Git is like Microsoft Word's "Track Changes" feature — the tool that records edits. GitHub is like OneDrive or Google Drive — the place where you store that file and share it with friends.

Part 2 — Under the Hood

The three stages of saving your work 📦

✏️

Working Directory

The files on your computer you're actively editing

→
📋

Staging Area

You choose exactly which changes to include in the next save

→
📦

Repository

A permanent saved snapshot, called a "commit"

In practice, that looks like three short commands:

git add turtle_square.py # move my change to staging git commit -m "Added a square-drawing function" # save a snapshot git push # send it up to GitHub

Don't worry about memorizing these yet — we'll practice them together with real code very soon!

Part 2 — Key Vocabulary

Words you'll hear constantly 🗂️

📦

Repository (repo)

A project folder that Git is tracking the history of

📸

Commit

A saved snapshot of your project at one point in time

🌿

Branch

A separate timeline to try new ideas safely

👯

Clone

Downloading your own copy of a repo to your computer

⬆️

Push

Sending your saved commits up to GitHub

⬇️

Pull

Downloading the latest commits from GitHub

🔀

Merge

Combining a branch's changes back into the main project

☁️

Remote

The online copy of your repo — usually on GitHub
